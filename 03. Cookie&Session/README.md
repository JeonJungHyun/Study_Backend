# 🍪 03. Cookie & Session 

<details>
<summary>1. 쿠키와 세션은 왜 필요한가?</summary> </br>

웹 브라우저와 웹 서버가 데이터를 주고 받을 때 HTTP 통신을 이용하는데, HTTP는 `Stateless(무상태성)` 프로토콜이다. </br>

즉, 서버는 `이전 요청의 상태를 저장하지 않는다는 것이다.` </br>

따라서 한 번 작성하거나 사용한 뒤에는 반복적으로 작업하지 않도록 저장해 두는 것이 필요한데, </br>

이때 등장하는 개념이 바로 쿠키와 세션이다. </br>

</details>

<details>
<summary>2. Cookie</summary> </br>

❶ Cookie란? </br>

쿠키란 사용자가 어떤 웹 사이트에 들어갔을 때 사용자 브라우저에 저장되는 작은 데이터이며, `키와 값(key, value)` 형태로 구성되어 있다. </br>

서버는 응답 시 `Set-Cookie` 헤더를 통해 쿠키를 생성한다. </br>

브라우저는 쿠키를 저장해 놓았다가, 사용자가 요청하면 해당 정보를 함께 보내서 서버가 사용자를 식별할 수 있게 해준다. </br>

</br>

❷ 전송 방식 </br>

쿠키를 사용하여 로그인하는 과정을 살펴보면, 아래와 같다. </br>

<img width="820" height="396" alt="image" src="https://github.com/user-attachments/assets/8ec91fbc-f814-46e8-9130-b32c2098c508" />

1) 클라이언트가 서버에 요청을 보낸다. </br>
2) 서버가 응답과 함께 쿠키를 생성한다. </br>
3) 브라우저가 쿠키를 저장한다. </br>
4) 이후 요청마다 쿠키가 자동 전송된다. </br>

</br>

❸ 주로 사용되는 곳 </br>

- 로그인 상태 유지 (아이디 기억하기, 자동 로그인) </br>

- 장바구니 (로그인하지 않아도 상품 유지) </br>

- 사용자 설정 저장 (다크모드 여부, 팝업 "오늘 하루 보지 않기" 등) </br>

</br>

❹ 쿠키의 한계 </br>

- 보안 취약성: 클라이언트에 저장되므로 사용자가 임의로 수정하거나 탈취하기 쉽다. </br>

- 용량 제한: 하나의 쿠키는 4KB로 크기가 제한되어 있다. </br>

- 성능 저하: 매 요청마다 서버로 전송되기 때문에 쿠키가 많아지면 네트워크 부하가 커진다. </br>

</details>

<details>
<summary>3. Session</summary> </br>

❶ Session이란? </br>

세션은 `서버에 저장되는 정보`이며, 사용자의 민감한 정보는 서버 측에서 관리하고 </br>

클라이언트에게는 그 정보를 찾을 수 있는 '열쇠'인 **세션 ID(Session ID)** 만 발급해 주는 방식이다. </br>

</br>

❷ 동작 방식 </br>

1) 클라이언트가 서버에 로그인(인증) 요청을 보낸다. </br>

2) 서버는 계정 정보를 확인 후, 서버의 세션 저장소(메모리 또는 DB)에 정보를 저장하고 고유한 Session ID를 발급한다. </br>

<img width="4590" height="1247" alt="image" src="https://github.com/user-attachments/assets/435e56b7-575e-4066-89b3-a6cb1c15c4e4" />

3) 서버는 응답 헤더의 Set-Cookie에 Session ID를 담아 보낸다. </br>

4) 클라이언트는 이후 요청마다 이 Session ID가 담긴 쿠키를 함께 전송한다. </br>

5) 서버는 전달받은 ID로 세션 저장소에서 사용자 정보를 확인하여 응답한다. </br>

<img width="4590" height="1247" alt="image" src="https://github.com/user-attachments/assets/a9691ae0-95b6-4cb7-8d37-efe3894c06db" />

</br>

❸ 장단점 </br>

- 장점: 실제 데이터가 서버에 있으므로 쿠키보다 훨씬 안전하다. </br>

- 단점: </br>

  - 사용자가 많아질수록 서버 메모리(RAM) 사용량이 늘어나 `서버 부하가 발생할 수 있다.` </br>
  
  - 서버 재시작 시 메모리에 저장된 세션 정보가 휘발될 수 있다. (이를 방지하기 위해 Redis 같은 외부 저장소를 사용한다.) </br>    

</details>

<details><summary>4. 쿠키의 주요 보안 옵션 (보안 강화)</summary> </br>
 
쿠키는 브라우저라는 공개된 장소에 저장되기 때문에 해킹에 매우 취약하다. </br>

이를 방지하기 위해 백엔드 개발자는 쿠키를 생성(Set-Cookie)할 때 반드시 아래의 보호막들을 설정해야 한다. </br>

</br>

❶ HttpOnly (자바스크립트 접근 차단) </br>

브라우저의 자바스크립트(document.cookie)가 이 쿠키를 읽거나 수정하는 것을 원천적으로 차단하는 옵션이다. </br>

Q : 왜 쓸까? </br>

A : 해커가 게시판 등에 악성 스크립트를 심어 사용자의 쿠키를 탈취하는 XSS(Cross-Site Scripting) 공격을 시도할 때, </br>
이 옵션이 켜져 있으면 스크립트가 세션 ID를 읽어갈 수 없어 사용자의 계정을 탈취하는 것을 막을 수 있다. </br>

</br>

❷ Secure (암호화 통신 강제) </br>

`HTTPS(보안 연결) 프로토콜을 사용하는 암호화된 통신 환경에서만` 쿠키를 주고받도록 제한하는 설정이다. </br>

Q : 왜 쓸까? </br>

A : 일반 HTTP 통신은 데이터가 암호화되지 않은 평문으로 전달된다. </br>
만약 공용 와이파이 같은 곳에서 통신 데이터를 가로채면 쿠키 내용이 그대로 노출되는데, </br>
Secure 옵션은 암호화된 통로로만 쿠키가 다니게 하여 보안을 강화한다. </br>

</br>

❸ SameSite (사이트 간 요청 위조 방어) </br>

쿠키를 생성한 도메인과 현재 요청을 보내는 도메인이 일치할 때만 쿠키를 전송하도록 제한한다. </br>

Q : 왜 쓸까? </br>
A : 사용자가 의도치 않게 공격용 링크를 클릭했을 때, 본인의 권한(쿠키)을 이용해 서버에 잘못된 요청(예: 비밀번호 변경, 송금 등)을 </br>
보내게 만드는 CSRF(Cross-Site Request Forgery) 공격을 방지하기 위한 현대적인 필수 설정이다. </br>

</details>

<details><summary>5. 세션(Session)의 한계와 서버 확장성 문제 (Scalability)</summary> </br>
  
❶ 서버 메모리(RAM) 부하 문제 </br>

세션 정보는 기본적으로 서버의 메모리에 저장된다. </br>

동시 접속자가 수만 명으로 늘어나면 서버 메모리를 과도하게 점유하게 되어 전체적인 시스템 성능이 저하되거나, 최악의 경우 서버가 다운될 수 있다. </br>

</br>

❷ 세션 불일치 문제 (서버가 여러 대일 때) </br>

트래픽이 많아져서 서버를 2대(서버 A, 서버 B)로 늘린 상황을 가정해 보자. </br>

사용자가 서버 A에서 로그인하여 세션이 생성되었다. (서버 A의 메모리에 저장됨) </br>

로드밸런서(트래픽 분산 장치)가 다음 요청을 서버 B로 보낸다. </br>

서버 B의 메모리에는 해당 사용자의 세션 정보가 없다. </br>

👉 결과 : 사용자는 분명 로그인을 했음에도 불구하고, 다음 페이지를 누르자마자 다시 로그인을 하라는 메시지를 받게 된다. </br>

</br>

❸ 실무에서의 해결 방법 (External Session Store) </br>

In-Memory DB 사용 (Redis): 각 서버의 개별 메모리에 저장하지 않고, 

모든 서버가 공통으로 접근할 수 있는 아주 빠른 외부 저장소인 Redis 등에 세션 정보를 따로 모아서 관리한다. </br>

어떤 서버로 요청이 가든 공용 저장소만 확인하면 되므로 세션 불일치 문제가 완벽하게 해결된다. </br>

</details>

<details>
<summary>6. 세션의 관리: 만료와 파기 (Lifecycle)</summary> </br>
 
세션은 서버의 유한한 자원을 사용하므로 효율적인 관리가 필수이다. </br>
  
- 세션 타임아웃(Timeout) </br>

: 사용자가 마지막 요청을 보낸 시점부터 특정 시간(보통 30분) 동안 아무런 요청이 없으면, </br>
서버는 해당 세션을 "만료"된 것으로 간주하고 삭제한다. 이를 통해 활동하지 않는 사용자가 메모리를 계속 차지하는 것을 방지한다. </br>

- 로그아웃의 원리</br>

: 사용자가 로그아웃을 누르면, 백엔드 서버는 세션 저장소(메모리나 Redis)에서 해당 세션 ID를 즉시 삭제한다. </br>
동시에 브라우저에게 "이 쿠키는 이제 만료되었으니 삭제하라"는 응답을 보냄으로써 양쪽의 정보를 모두 파기한다. </br>

</details>
