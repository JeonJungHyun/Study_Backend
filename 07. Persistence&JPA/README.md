# 🔄 07. Persistence & JPA

<details>
<summary>1. 영속성(Persistence)이란? </summary> </br>

우리가 자바 코드로 만든 객체는 프로그램이 꺼지면 메모리에서 순식간에 사라진다. </br>
하지만 회원 정보나 게시글 같은 데이터가 사라지면 서비스 운영이 불가능하다. </br>
영속성(Persistence)은 "데이터를 실행 중인 프로그램 밖에서도 사라지지 않게 저장소(DB)에 유지하는 성질". </br>
즉, 메모리에 떠다니는 데이터를 딱딱한 비석(하드디스크/DB)에 새겨서 영구히 보존하는 것을 말한다. </br>
이 과정이 있어야 서버를 껐다 켜도 사용자의 소중한 데이터가 그대로 남아있을 수 있다.

</details>

<details>
<summary>2. ORM과 JPA </summary> </br>
자바는 '객체' 중심이고, DB는 '표(Table)' 중심이라 서로 노는 방식이 완전히 다르다. </br>
이 차이 때문에 예전에는 개발자가 중간에서 엄청난 양의 SQL 문을 직접 받아쓰기하듯 짜야 했다. </br>
객체 관계형 매핑(Object-Relational Mapping, ORM)이란 프로그래밍 언어와 데이터베이스 간에 호환되지 않는 데이터를 변환하는 기법이다. </br>
변환 과정 중, ORM 기술은 프로그래밍 언어 내에서 사용될 수 있는 가상 객체 데이터베이스를 생성한다. </br>
그 결과, ORM에서는 VO 객체가 가지고 있는 정보가 개별적인 데이터베이스가 아닌 java.util.Map과 흡사한 컬렉션에 담겨 관리된다고 볼 수 있다. </br>
따라서 관계형 데이터베이스에 접속할 때 쓰이는 SQL을 따로 작성할 필요가 없음은 물론, 아래와 같은 특징을 보인다. </br>

</br>

  1. SQL을 개발자가 직접 작성할 수고를 덜어주며, SQL처럼 절차적인 코드 대신 객체 지향적인 코드로 작동해 비즈니스 로직에 집중할 수 있게 해 준다. </br>
  2. ORM은 독립적으로 구현되며 해당 객체를 재활용할 수 있어 재사용성이 높고 유지 보수가 편리하다. </br>
  3. DBMS에 대한 종속성이 줄어든다. </br>
  4. 자바 ORM의 대표적인 예로는 오픈 소스 프레임워크인 하이버네이트가 있다. </br>

✏️ ORM(Object-Relational Mapping): "객체랑 테이블을 1:1로 매핑(연결)해줄게!" 하는 기술. </br>
✏️ JPA(Java Persistence API): 자바 진영에서 정한 ORM 기술의 공식 표준 설계도(인터페이스). </br>

🌟 핵심 포인트: JPA는 "이렇게 데이터를 관리하자"라는 약속일 뿐이고, </br>
실제로 그 약속을 지키며 뒤에서 SQL을 만들어 일하는 진짜 일꾼(구현체)은 `Hibernate(하이버네이트)`라는 녀석이다.

</details>

<details>
<summary>3. 영속성 컨텍스트(Persistence Context) </summary> </br>

JPA는 데이터를 DB에 바로 던지지 않고, 중간에 `'영속성 컨텍스트'`라는 가상의 창고에 먼저 담아 관리한다. </br>

1️⃣ 1차 캐시: 한 번 읽어온 데이터는 창고에 보관한다. 똑같은 걸 또 찾으면 DB까지 안 가고 창고에서 바로 꺼내준다. </br>

2️⃣ 변경 감지(Dirty Checking): 창고에 담긴 객체의 값만 살짝 바꾸면, JPA가 처음 상태랑 비교해서 "어? 바뀌었네?" 하고 알아서 업데이트 SQL을 날려준다. </br>
따로 update() 코드를 짤 필요가 없다. </br>

3️⃣ 쓰기 지연: SQL을 쌓아두었다가 트랜잭션이 끝나는 시점(또는 flush() 호출 시)에 한꺼번에 DB로 쏜다. 이를 통해 네트워크 통신 횟수를 줄여 성능을 높인다. </br>

</details>

<details>
<summary>4. Entity vs DTO </summary> </br>

백엔드 개발을 하다 보면 이 두 개념을 확실히 나눠야 사고가 안 난다. </br>

- Entity (@Entity): DB 테이블과 직접 연결된 '데이터의 본체'. 창고(영속성 컨텍스트)의 관리를 받는 주인공이다. </br>
- DTO (Data Transfer Object): 데이터를 여기저기 배달할 때 쓰는 '일회용 가방'. </br>

🤔 왜 나눠서 써? </br>
DB 원본인 Entity는 아주 소중해서 함부로 화면(API)에 노출하면 보안상 위험하고 구조가 꼬이기 쉽다. </br>
그래서 밖으로 데이터를 보낼 때는 필요한 것만 가방(DTO)에 담아서 보내는 게 정석이다.

</details>

<details>
<summary>5. JPA vs MyBatis : 자동화 vs 수동 관리</summary> </br>

DB와 대화하는 두 가지 대표적인 방식이다. </br>

- MyBatis: "내가 직접 SQL을 예쁘게 짜서 DB에 줄게!" (수동 / SQL 중심) </br>

- JPA: "난 자바 객체만 잘 만들게, SQL은 네가 알아서 짜줘!" (자동 / 객체 중심) </br>


🌟 요즘 트렌드: </br>

예전에는 MyBatis를 많이 썼지만, 현대적인 백엔드 개발에서는 유지보수가 편하고 개발 속도가 압도적으로 빠른 JPA가 대세다. </br>
다만, 복잡한 통계 쿼리나 아주 세밀한 SQL 최적화가 필요한 경우에는 개발자가 직접 SQL을 제어하는 MyBatis가 더 유리할 수도 있다. </br>
최근에는 기본 로직은 JPA로, 복잡한 쿼리는 Querydsl이나 Native SQL로 보완하는 방식을 많이 쓴다. </br>

</details>

<details>
<summary>6. 정리 </summary> </br>

- EntityManager (em): 영속성 컨텍스트(창고)를 관리하는 총책임자 매니저. </br>
- @Entity: "이 클래스는 DB 테이블이랑 연결될 거야!"라고 붙여주는 이름표. </br>
- Spring Data JPA: JpaRepository라는 치트키를 써서 코딩 없이 DB 조작을 끝내주는 도구. </br>

</details>

<details>
<summary>🔥 핵심 면접 질문 리스트 🔥 </summary></br>

**Q1. JPA(Java Persistence API)와 Hibernate의 관계는 무엇인가요?** </br>

답변: "JPA는 자바 진영에서 '객체와 DB를 이렇게 연결하자'라고 정해놓은 **인터페이스(설계도)** 이고, </br>
Hibernate는 그 설계도를 바탕으로 실제로 구현하여 동작하게 만드는 **구현체(일꾼)** 입니다.

우리는 JPA라는 표준 규격에 맞춰 코드를 짜면, 실제 실행은 Hibernate가 뒤에서 SQL을 생성하며 수행해 주는 구조입니다." </br>

</br>

**Q2. 영속성 컨텍스트(Persistence Context)의 이점 3가지만 설명해 주세요.** </br>

답변: "첫째, 1차 캐시를 통해 동일한 데이터를 조회할 때 DB 접근을 줄여 성능을 높입니다. </br>

둘째, **변경 감지(Dirty Checking)** 로 객체의 값만 바꿔도 자동으로 업데이트 SQL을 생성해 줍니다. </br>

셋째, 쓰기 지연 기능을 통해 여러 SQL을 모았다가 트랜잭션이 끝날 때 한꺼번에 실행하여 네트워크 비용을 절감합니다." </br>

</br>

**Q3. '변경 감지(Dirty Checking)'가 일어나는 원리는 무엇인가요?** </br>

답변: "JPA가 영속성 컨텍스트에 엔티티를 처음 보관할 때, 그 당시의 상태를 복사해서 **'스냅샷'** 으로 찍어둡니다. </br>

트랜잭션이 끝나는 시점에 현재 엔티티의 상태와 이 스냅샷을 비교하여, </br>
값이 변경된 부분이 있다면 자동으로 UPDATE SQL을 생성해서 DB에 반영합니다." </br>

</br>

**Q4. DB 테이블과 매핑된 Entity를 직접 API 응답으로 내보내면 안 되는 이유는?** </br>
 
답변: "보안과 유연성 때문입니다. </br>

Entity에는 비밀번호나 내부 로직용 데이터 등 노출되면 안 되는 정보가 포함될 수 있고, DB 구조가 API 스펙에 직결되어 버립니다. </br>

만약 DB 컬럼명 하나만 바꿔도 모든 API가 깨지게 되므로, 외부에 데이터를 전달할 때는 반드시 전용 가방인 DTO로 변환해서 보내야 합니다." </br>

</br>

**Q5. JPA와 MyBatis 중 어떤 상황에 무엇을 선택하는 것이 좋을까요?** </br>

답변: "대부분의 현대적인 비즈니스 로직에는 생산성과 객체 지향적 설계가 유리한 JPA를 선택하는 것이 대세입니다. </br>

하지만 복잡한 통계 쿼리나 DB 성능 최적화가 극도로 중요한 경우에는 SQL을 직접 제어하는 MyBatis가 유리할 수 있습니다. </br>

실무에서는 주로 JPA를 기본으로 사용하되, 복잡한 쿼리는 Querydsl이나 Native SQL을 섞어서 보완합니다." </br>

</br>

**Q6. JpaRepository 인터페이스만 만들었는데 어떻게 DB 조작이 가능한 건가요?** </br>

답변: "Spring Data JPA가 런타임에 해당 인터페이스를 보고 동적으로 구현 객체를 생성해서 주입해 주기 때문입니다. </br>

우리는 메서드 이름만 규칙에 맞게 지으면(예: findByUsername), </br>
스프링이 그 이름을 분석해서 적절한 JPQL이나 SQL을 자동으로 생성하여 실행해 줍니다." </br>

</br>

</details>
