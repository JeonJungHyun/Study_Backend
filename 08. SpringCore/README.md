# 🍃 08. Spring Core (프레임워크의 심층 분석)

<details>
<summary>1. IoC (Inversion of Control: 제어의 역전) </summary> </br>

❶ 주도권의 완전한 이양 </br>

전통적인 프로그래밍(순수 자바)에서는 개발자가 A 객체 안에서 B b = new B();를 직접 생성한다. </br>
이렇게 되면 A는 B가 없으면 존재할 수 없는 '강한 결합(Tight Coupling)' 상태가 된다. </br>
만약 B를 C로 바꾸려면 A의 소스 코드를 직접 열어서 수정해야 하며, 이는 유지보수의 재앙으로 이어진다. </br>

</br>

❷ IoC 컨테이너의 역할 (ApplicationContext) </br>

스프링은 이 문제를 해결하기 위해 객체의 생성과 관리 권한을 프레임워크가 가져간다. </br>

- 객체 관리의 자동화 </br>
: 설정 정보(Annotation 등)를 바탕으로 객체를 스스로 생성하고, 의존성을 주입하며, 소멸까지 책임지는 '지능형 공장' 역할을 한다. </br>

</br>

- ApplicationContext의 계층 구조 </br>
: 단순히 객체만 담는 BeanFactory를 상속받아, 국제화(i18n), 이벤트 프로퍼티 로딩 등 엔터프라이즈 환경에 필요한 모든 부가 기능을 통합 관리한다. </br>

</br>

✏️ 결론: 개발자는 "어떻게 객체를 만들지" 고민하지 않고, 오직 서비스의 "비즈니스 로직"에만 집중할 수 있게 된다. </br>

</details>

<details>
<summary>2. DI (Dependency Injection: 의존성 주입) </summary> </br>

❶ 전략적인 부품 조립 </br>

한 클래스가 자신의 기능을 수행하기 위해 다른 클래스의 메서드를 호출하는 상태를 '의존한다'고 한다. </br>
DI는 이 '필요한 객체'를 내가 직접 만드는 게 아니라, 외부(컨테이너)에서 주입(전달)받는 것을 의미한다. </br>

</br>

❷ 생성자 주입(Constructor Injection)이 '절대적 표준'인 이유 </br>

1️⃣ 객체의 불변성(Immutability) </br>
: 필드를 final로 선언할 수 있다. 한 번 주입된 의존성은 앱이 꺼질 때까지 변하지 않음을 보장하므로 멀티스레드 환경에서 안전하다. </br>

</br>

2️⃣ 순환 참조(Circular Dependency) 즉시 발견 </br>
: A가 B를, B가 A를 참조할 경우, 서버가 뜨는 시점에 BeanCurrentlyInCreationException을 발생시켜 설계를 수정하도록 유도한다. </br>
(Setter 주입은 실제 호출 전까지 에러를 잡지 못해 위험하다.) </br>

</br>

3️⃣ 테스트 용이성 </br>
: 스프링 프레임워크 없이 new 키워드만으로 테스트 코드를 짤 때, 필요한 가짜 객체(Mock)를 생성자로 직접 넣어줄 수 있어 단위 테스트가 매우 정교해진다. </br>

</details>

<details>
<summary>3. Spring Bean & Singleton (스프링 빈과 싱글톤의 심연) </summary> </br>

❶ 싱글톤(Singleton)의 존재 이유: 메모리 성능 최적화 </br>

웹 서비스는 수만 명의 사용자가 동시에 접속한다. </br>
만약 유저 한 명당 Service 객체를 하나씩 생성한다면(초당 1,000개 생성/삭제), 자바의 가비지 컬렉터(GC)가 과부하되어 서버가 멈출 수 있다. </br>
스프링은 객체를 딱 하나만 만들어 공유함으로써 GC의 부담을 줄이고 메모리를 획기적으로 아낀다. </br>

</br>

❷ 무상태성(Stateless)과 동시성 제어 </br>

싱글톤 객체는 여러 스레드(사용자)가 동시에 공유하므로 Heap vs Stack 개념이 중요하다. </br>

- Heap 영역 (위험) </br>
: 객체의 멤버 변수(필드)가 저장되는 공유 공간이다. 여기에 사용자 데이터를 쓰면 데이터가 꼬이는 동시성 문제가 발생한다. </br>

</br>

- Stack 영역 (안전) </br>
: 메서드가 호출될 때마다 각 사용자 전용으로 생기는 독립 공간이다. </br>
지역 변수는 이 스택 영역에만 존재하므로, 수만 명이 동시에 접속해도 데이터가 절대 섞이지 않는다. </br>

</br>

🌟 결론: 싱글톤 빈 내부에는 상태를 저장하는 공유 변수(필드)를 절대로 만들지 말아야 한다. </br>

</details>

<details>
<summary>4. AOP (Aspect Oriented Programming: 관점 지향 프로그래밍) </summary> </br>

❶ 횡단 관심사(Cross-cutting Concerns)의 분리 </br>

비즈니스 로직(로그인, 주문)마다 공통적으로 들어가는 코드(로그, 트랜잭션, 보안)를 **'횡단 관심사'** 라고 한다. </br>
이를 별도의 모듈(Aspect)로 빼내어 관리하는 것이 AOP이다. </br>

</br>

❷ 프록시(Proxy) 매커니즘의 실체 </br>
스프링은 사용자가 부르는 '진짜 객체' 대신 **'가짜 대리인(Proxy) 객체'** 를 메모리에 올린다. </br>
사용자가 메서드를 호출하면 가짜 객체가 요청을 가로챈다. </br>
가짜 객체는 @Transactional 같은 설정을 확인해 DB 연결이나 로그 시작을 먼저 처리한다. </br>
그 후 진짜 객체의 로직을 실행하고, 끝나면 가짜 객체가 다시 결과를 받아 마무리 작업을 수행한다. </br>

</br>

✅ 결과: 개발자는 복잡한 공통 코드를 반복할 필요 없이 @Transactional 어노테이션 하나로 핵심 로직에만 집중할 수 있다. </br>

</details>

<details>
<summary>5. 스프링 빈 라이프사이클 (상세 프로세스) </summary> </br>

빈이 탄생하고 소멸하기까지 스프링 내부에서는 정교한 단계가 진행된다. </br>

1️⃣ 스프링 컨테이너 생성: 설정 정보(Java Config, Annotation)를 로딩하여 빈 정의(BeanDefinition)를 만든다. </br>

2️⃣ 스프링 빈 생성: 등록된 클래스를 기반으로 객체를 인스턴스화한다. (기본 싱글톤) </br>

3️⃣ 의존관계 주입: @Autowired 등을 통해 필요한 다른 빈들을 연결한다. </br>

4️⃣ 초기화 콜백: @PostConstruct 실행. 객체가 모든 주입을 마치고 사용 준비가 끝났음을 알린다. </br>

5️⃣ 사용: 실제 서비스 로직을 수행하며 요청을 처리한다. </br>

6️⃣ 소멸전 콜백: @PreDestroy 실행. 서버 종료 전 DB 커넥션 종료나 네트워크 해제 등 리소스를 정리한다. </br>

7️⃣ 스프링 종료: 컨테이너가 닫히며 모든 빈이 파괴된다.

</details>

<details>
<summary>6. 최종 정리: 왜 스프링인가? </summary> </br>

- IoC/DI: 객체 간의 결합도를 낮춰 코드 수정 없이 부품을 갈아 끼울 수 있는 유연성을 제공한다. </br>

- Singleton: 대규모 트래픽 환경에서 메모리 낭비를 방지하고 성능을 최적화한다. </br>

- AOP: 중복되는 부가 기능을 한곳에서 관리하여 코드 가독성과 유지보수성을 극대화한다. </br>

</br>

🌟 정리: 스프링은 좋은 객체 지향 설계를 할 수 있도록 도와주는 강력한 **'도구 상자'** 이자 **'관리자'** 이다.

</details>
