# 🍃 08. Spring Core (프레임워크의 심층 분석)

<details>
<summary>1. IoC (Inversion of Control: 제어의 역전) - "사장이 직접 안 움직인다" </summary> </br>

❶ 전통적인 방식 (개발자가 노가다 할 때) </br>

내가 햄버거 가게 사장인데, 감자튀김이 필요할 때마다 직접 시장 가서 감자 사오고 씻고 튀기는 방식이다. </br>

🤔 문제: 감자 파는 아저씨(B)가 장사를 접으면, 사장(A)인 나도 햄버거를 못 만든다. 이게 **'강한 결합'** 이다. </br>

</br>

❷ IoC 방식 (스프링 컨테이너가 해줄 때) </br>

사장은 가만히 있고, **'주방 매니저(스프링)'** 를 고용한다. 사장은 매니저한테 "감자튀김 하나 내놔!"라고 말만 한다. </br>
매니저가 감자를 어디서 구해오든, 어떻게 튀기든 사장은 상관없다. </br>

- 객체 관리의 자동화: 스프링이 알아서 객체를 만들고, 연결하고, 버려준다. </br>

✏️ 결론: 개발자는 객체를 만드는 귀찮은 일에서 해방되어, 진짜 중요한 **'기능 구현(레시피)'** 에만 집중하게 된다. </br>

</details>

<details>
<summary>2. DI (Dependency Injection: 의존성 주입) - "부품은 밖에서 꽂아주는 것" </summary> </br>

❶ 의존성이란? </br>

컴퓨터가 본체만 있으면 안 되고 **'모니터'** 가 꼭 있어야 화면이 나오는 것처럼, 한 클래스가 일하기 위해 다른 클래스를 꼭 필요로 하는 상태다. </br>

</br>

❷ 왜 생성자에 꽂아주는 게(생성자 주입) 최고인가? </br>

컴퓨터를 살 때, 모니터를 본체 살 때 딱 같이 꽂아서 사는 방식이다. </br>

</br>

1️⃣ 절대 안 바뀜: 한 번 꽂아두면 컴퓨터 쓰는 도중에 모니터가 빠질 일이 없다. (안전함) </br>

</br>

2️⃣ 미리 체크: 만약 모니터가 고장 났으면, 컴퓨터 전원을 켤 때 바로 삑삑 소리가 난다. </br>
나중에 게임하다가 꺼지는 것보다 훨씬 낫다. (에러 미리 발견) </br>

</br>

3️⃣ 테스트 굿: 모니터가 진짜인지 가짜인지 사장이 직접 바꿔 끼워보면서 테스트하기가 아주 쉽다. </br>

</details>

<details>
<summary>3. Spring Bean & Singleton (스프링 빈과 싱글톤) - "공유 경제" </summary> </br>

❶ 싱글톤: "프린터기는 한 대면 충분해" </br>

회사에 직원이 100명이라고 프린터기를 100대 살 필요가 없다. 한 대만 사서 다 같이 쓰는 게 돈이 절약된다. </br>
스프링도 똑같다. 요청이 1,000번 들어온다고 똑같은 객체를 1,000개 만들면 서버가 터진다. </br>
그래서 딱 하나만 만들어서 다 같이 쓴다. </br>

</br>

❷ 주의할 점: "공용 화장실에 내 짐 두지 마라" </br>

모두가 같이 쓰는 객체(싱글톤)인데, 거기다가 "나의 개인 데이터"를 저장하면 어떻게 될까? </br>
다른 사람이 내 데이터를 보거나, 내 데이터가 덮어씌워지는 대참사가 난다. </br>
그래서 **무상태(Stateless)** 가 중요하다. </br>
개인적인 데이터는 자기 주머니(메서드 안의 지역 변수)에만 넣어야 한다. </br>

</details>

<details>
<summary>4. AOP (관점 지향 프로그래밍) - "반복되는 건 기계가 하게 해라" </summary> </br>

❶ 횡단 관심사: "모든 음식에 들어가는 소금" </br>

로그인 체크, 로그 남기기, 트랜잭션... 이런 건 모든 기능에 다 들어간다. </br>
이걸 모든 코드에 일일이 다 적으려면 손목 나간다. </br>

</br>

❷ 프록시(Proxy) 매커니즘: "비서가 대신 처리해줌" </br>

사장이 결재 서류에 도장을 찍어야 하는데, 비서가 미리 서류 검토하고(로그인 체크), </br>
사장이 도장 찍으면(핵심 로직), 비서가 다시 서류 정리(로그 기록)를 하는 식이다. </br>

🌟 결과: 개발자는 도장 찍는(핵심 비즈니스 로직) 코드만 짜면 된다. </br>
나머지 귀찮은 일은 스프링이 비서(프록시)를 세워서 다 알아서 해준다. </br>

</details>

<details>
<summary>5. 스프링 빈 라이프사이클 (상세 프로세스) </summary> </br>

빈이 탄생하고 소멸하기까지 스프링 내부에서는 정교한 단계가 진행된다. </br>

</br>

1️⃣ 설정 로딩 (계획): @Service, @Component 같은 설정 정보를 읽어서 어떤 객체를 만들지 빈 정의(BeanDefinition) 명단을 작성한다.

"어떤 가구를 만들지 설계도를 쭉 훑어보는 단계."

</br>

2️⃣ 빈 생성 (탄생): 작성된 명단을 보고 클래스를 실체화하여 **객체(Instance)** 를 만든다. (기본적으로 싱글톤으로 생성된다.)

"설계도를 보고 일단 가구의 뼈대를 뚝딱뚝딱 만드는 단계."

</br>

3️⃣ 의존관계 주입 (조립): @Autowired가 붙은 곳을 찾아 필요한 다른 빈들을 연결해 준다. 즉, **DI(Dependency Injection)** 가 여기서 일어난다.

"완성된 가구 뼈대에 문 손잡이나 다리를 꽂아주는 부품 조립 단계."

</br>

4️⃣ 초기화 콜백 (준비 완료): **@PostConstruct**가 붙은 메서드를 실행한다. 모든 조립이 끝났으니 이제 사용해도 좋다는 신호를 보낸다.

"가구 조립이 끝났으니 마지막으로 먼지를 닦고 '사용 가능' 스티커를 붙이는 단계."

</br>

5️⃣ 사용 (열일): 실제 애플리케이션에서 비즈니스 로직을 수행하며 사용자의 요청을 처리한다.

"이제 손님이 와서 가구를 실제로 사용하는 전성기 단계."

</br>

6️⃣ 소멸전 콜백 (정리): 서버가 종료되기 직전, **@PreDestroy**가 붙은 메서드를 실행합니다. DB 연결을 끊거나 리소스를 반납한다.

"가게 문 닫기 전에 가구를 덮개로 덮고 주변을 정리하는 퇴근 준비 단계."

</br>

7️⃣ 스프링 종료 (퇴근): 컨테이너가 닫히면서 관리하던 모든 빈이 파괴된다.

"불 끄고 완전히 퇴근하며 객체가 메모리에서 사라지는 단계."
</details>

<details>
<summary>6. 최종 정리: 왜 스프링인가? </summary> </br>

- IoC/DI: 객체 간의 결합도를 낮춰 코드 수정 없이 부품을 갈아 끼울 수 있는 유연성을 제공한다. </br>

- Singleton: 대규모 트래픽 환경에서 메모리 낭비를 방지하고 성능을 최적화한다. </br>

- AOP: 중복되는 부가 기능을 한곳에서 관리하여 코드 가독성과 유지보수성을 극대화한다. </br>

</br>

🌟 정리: 스프링은 좋은 객체 지향 설계를 할 수 있도록 도와주는 강력한 **'도구 상자'** 이자 **'관리자'** 이다.

</details>

<details>
<summary>🔥 핵심 면접 질문 리스트 🔥 </summary></br>

**Q1. IoC(제어의 역전)란 무엇이며, 왜 필요한가요?** </br>

답변: "객체의 생성과 생명주기 관리 권한을 개발자가 아닌 스프링 컨테이너가 가져가는 것을 말합니다. </br>

개발자가 직접 new를 통해 객체를 만들면 객체 간의 결합도가 높아져 수정이 어렵지만, </br>
스프링이 대신 관리해주면 설정만으로 부품을 갈아 끼우듯 유연하게 코드를 짤 수 있어 유지보수성이 비약적으로 향상됩니다." </br>

</br>

**Q2. 의존성 주입(DI)의 방법 3가지 중 '생성자 주입'을 권장하는 이유는?** </br>

답변: "첫째, 객체의 불변성을 보장할 수 있습니다. 한 번 주입되면 런타임에 바뀔 일이 없어 안전합니다.

둘째, 누락 방지입니다. 객체를 생성할 때 필요한 부품이 없으면 컴파일 에러가 나므로 실수를 미리 방지합니다.

셋째, 순환 참조 문제를 애플리케이션 실행 시점에 바로 잡아낼 수 있어 가장 권장되는 방식입니다." </br>

</br>

**Q3. 스프링 빈(Bean)이란 무엇이며, 어떻게 등록하나요?** </br>

답변: "스프링 컨테이너가 관리하는 자바 객체를 **빈(Bean)** 이라고 부릅니다.

주로 @Component, @Service, @Repository, @Controller 같은 어노테이션을 클래스 위에 붙여서 자동으로 등록하거나, </br>
@Configuration 클래스 안에서 @Bean을 사용해 수동으로 등록할 수도 있습니다." </br>

</br>

**Q4. AOP(관점 지향 프로그래밍)를 한 문장으로 정의한다면?** </br>

답변: "여러 곳에서 공통적으로 쓰이는 '부가 기능(로깅, 트랜잭션 등)'을 핵심 비즈니스 로직에서 분리하여 한곳에서 관리하는 기술입니다. </br>

이를 통해 코드 중복을 제거하고, 개발자가 핵심 로직에만 집중할 수 있게 도와줍니다." </br>

</br>

**Q5. 빈 라이프사이클 중 @PostConstruct와 @PreDestroy는 언제 사용하나요?** </br>

답변: "@PostConstruct는 빈의 의존관계 주입이 모두 끝난 직후, 초기화 작업을 위해 실행됩니다.

@PreDestroy는 컨테이너가 종료되기 직전, DB 연결 종료나 파일 닫기 같은 정리 작업을 위해 사용합니다.

생성자 호출 시점에는 아직 의존성 주입이 완료되지 않았을 수 있기 때문에, 안전한 초기화를 위해 이 콜백 메서드들을 활용합니다." </br>

</br>

**Q6. 스프링 빈이 기본적으로 '싱글톤'인 이유는 무엇일까요?** </br>

답변: "서버 자원의 효율적 사용을 위해서입니다.

엔터프라이즈급 서비스는 수많은 요청이 동시에 들어오는데, 그때마다 객체를 새로 만들면 메모리 부하가 감당이 안 됩니다.

따라서 객체를 하나만 생성해 공유함으로써 메모리 낭비를 방지하고 성능을 최적화합니다." </br>

</br>

</details>
