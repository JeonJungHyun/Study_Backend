# 🔐 04. Authentication

<details><summary>1. 인증(Authentication)과 인가(Authorization)</summary> </br>

❶ 인증(Authentication) </br>

- 정의 : "당신은 누구입니까?"를 확인하는 절차이다. </br>

- 설명 : 인증은 시스템에 접근하려는 사용자가 주장하는 신원이 실제와 일치하는지 검증하는 절차이다. 쉽게 말해 `신원 확인` 단계. </br>

- 비유 : </br>
      호텔 프런트에 가서 예약 내역(ID)을 말하고 신분증(Credential)을 보여준다. </br>
      직원은 예약 명단과 신분증을 대조해 본다. </br>
      확인이 되면 객실 키(Key)를 발급해 준다. </br>

- 웹 환경에서의 인증 요소 : </br>

  - 지식 기반 (What you know): 비밀번호(Password), PIN 번호. </br>
  
  - 소유 기반 (What you have): 스마트폰(SMS/OTP), 보안 카드, 하드웨어 키(YubiKey). </br>
  
  - 생체 기반 (Who you are): 지문, Face ID, 홍채 인식. </br>

  </br>

❷ 인가 (Authorization) </br>

- 정의 : "당신은 이 일을 할 권한이 있습니까?"를 확인하는 절차이다. </br>

- 설명 : </br>

      인증을 마친 사용자가 특정 자원에 접근하거나 특정 기능을 수행할 수 있는지 허용하는 과정이다. 
      건물에 들어온 사람이 관리자 구역에 들어갈 수 있는지 보안 등급을 확인하는 것과 같다.
      
- 비유 : </br>

      객실 키(인증 완료된 증표)를 받아서 엘리베이터에 탔다. 
      내 방이 있는 10층 버튼은 눌리지만, VIP 라운지인 20층 버튼은 눌리지 않는다. 
      직원이 내가 누군지는 알지만(인증 O), 20층에 갈 권한은 주지 않았기(인가 X) 때문이다. 

- 웹 환경에서의 인가 예시 : </br>

  - 일반 사용자는 GET /api/posts (글 조회)는 가능하지만, DELETE /api/posts/1 (글 삭제)는 불가능함.
    
  - 유튜브 프리미엄 회원은 ‘백그라운드 재생’ 기능을 사용할 권한이 있음.
  
 
🌟 백엔드의 흐름: 모든 요청은 먼저 인증을 거쳐 누구인지 파악한 뒤, 그 사람이 가진 권한에 따라 인가 처리를 하여 데이터를 줄지 말지 결정한다. </br>

</details>

<details><summary>2. JWT </summary> </br>

❶ JWT란 무엇인가? </br>

JWT는 인증에 필요한 정보를 JSON 객체에 담아 암호화한 전자 서명된 토큰이다. </br>

가장 큰 특징은 서버가 상태를 저장하지 않는(Stateless) 방식이라는 점이다. </br>

- 세션: 서버가 "이 번호표(ID) 가진 사람 누구더라?" 하고 장부를 뒤진다. </br>

- JWT: 서버가 신분증에 직인을 찍어 사용자에게 준다. 사용자가 신분증을 보여주면 서버는 자기 직인이 맞는지 확인만 한다. </br>

</br>

❷ JWT의 3단계 구조 (점(.)으로 연결된 세 부분) </br>

JWT는 Header.Payload.Signature로 구성되며, 각 부분은 누구나 볼 수 있게 Base64로 인코딩되어 있다. </br>

마치aaaaaaa.bbbbbbb.ccccccc 와 같은 형태이다. </br>

1. Header (헤더): “봉투” </br>
토큰의 타입과 암호화 알고리즘 정보를 담고 있다. `이 토큰을 어떻게 검증해야 하는지` 알려주는 설명서이다. </br>

`
{
  "alg": "HS256", // 서명에 사용할 알고리즘 (HMAC SHA256) 
  "typ": "JWT"    // 토큰 타입
} 
`

</br>

2. Payload (페이로드): “내용물” </br>

우리가 실제로 전달하고 싶은 데이터. 실제 유저의 아이디, 권한, 토큰 만료 시간 등이 담긴다. </br>

즉 클레임(Claim)들이 들어있는 곳이다. </br>

각 데이터 조각을 `클레임(Claim)`이라고 부른다. </br>

⚠️ 주의 </br>
JWT 는 암호화 된 것이 아니라 인코딩(Encoding)된 것이다. </br>

Base64 로 풀면 누구나 내용을 볼 수 있으므로, 비밀번호나 주민번호 같은 민감 정보는 절대 넣으면 안 된다. </br>

</br>

3. Signature (서명): “봉인 왁스” </br>

JWT 의 핵심이자, 위변조를 막는 안전장치이다. 이 부분은 헤더 + 내용 + 서버만 아는 비밀 키(Secret Key)를 조합해 만든 암호문이다. </br>

서버는 이 서명을 통해 토큰이 중간에 조작되지 않았음을 검증한다. </br> 

<img width="1024" height="559" alt="image" src="https://github.com/user-attachments/assets/435765d8-3a66-42b6-9131-9e53e97c4e06" />

</br>

❸ 어떻게 위조를 잡아내는가? (무결성 검증) </br>

서버가 클라이언트로부터 JWT 를 받았을 때, DB 조회 없이도 이 토큰을 믿을 수 있는(Stateless) 이유는 바로 서명(Signature) 때문이다. </br>

1. 해커의 시나리오: 해커가 Payload 의 role: "USER"를 role: "ADMIN"으로 몰래 고쳐서 보냈다고 가정해 보자. </br>

2. 서버의 검증: 서버는 받은 토큰의 Header와 (조작된) Payload를 가져옵니다. 여기에 서버가 가진 진짜 비밀 키를 섞어서 서명을 직접 다시 계산해 본다. </br>

3. 결과: 해커는 비밀 키를 모르기 때문에, 조작된 Payload 에 맞는 올바른 서명을 만들어낼 수 없다. </br>

결국 서버가 계산한 서명과 토큰에 적힌 서명이 다르게 나온다. </br>

서버는 즉시 “이건 위조된 토큰이야!” 하고 요청을 거부한다.</br>

</details>

<details><summary>3. JWT 인증 프로세스 (데이터의 흐름)</summary> </br>

1️⃣ 로그인 요청 (Client → Server) </br>

사용자가 아이디와 비밀번호를 입력하여 서버에 보낸다. 여기까지는 세션 방식과 똑같다.</br>
`
POST /login
Body: { id: "user1", pw: "1234" } ` </br>

 </br>

2️⃣ 검증 및 토큰 발급 (Server) </br>

서버는 DB 에서 계정 정보를 확인한다(인증). </br>

- 차이점: 계정이 확인되면 서버는 세션 저장소(Redis)에 아무것도 저장하지 않는다. </br>

대신, 사용자 정보(Payload)에 `비밀 키로 서명(Signature)을 하여 JWT(토큰)를 생성한다.` </br>

- 생성된 토큰: eyJhbG... (이 안에 “user1 / Admin” 정보가 다 들어있다) </br>

 </br>

3️⃣ 응답 (Server → Client) </br>

생성된 토큰을 클라이언트에게 전달한다. 보통 HTTP 응답의 Body나 Header에 담아 보낸다. </br>

`Response Body: { "accessToken": "eyJhbG..." }` </br>

 </br>

4️⃣ 저장 (Client) </br>  

클라이언트는 전달받은 토큰을 자신의 저장소에 보관한다. </br>

 </br>

5️⃣ 요청 (Client → Server) </br>

이제 사용자가 ‘내 정보 조회’ 같은 보호된 API 를 호출할 때, 저장해 둔 토큰을 꺼내 HTTP 헤더에 실어 보낸다. </br>

이때 Bearer 라는 인증 타입을 주로 사용한다. </br>

`GET /api/profile
Authorization: Bearer eyJhbG...`

 </br>
 
6️⃣ 검증 (Server) </br>

서버는 요청 헤더에서 토큰을 꺼낸다. </br>

- DB 조회 X: 서버는 DB 나 Redis 를 호출하지 않는다. </br>

- 서명 확인 O: 오직 자신이 가진 비밀 키(Secret Key)를 사용해 이 토큰의 서명이 유효한지 수학적으로 계산한다. </br>

🙆‍♀️ 서명이 맞다면? → “아, 내가 발급했던 토큰이 맞군. 유효기간도 안 지났네. 통과!” </br>
🙅‍♀️ 서명이 틀리다면? → “누가 조작했거나 가짜 토큰이군. 거절(401)!” </br>

</br>

7️⃣ 응답 (Server → Client) </br>

검증이 통과되면, 토큰의 Payload 에 들어있던 정보(userId: user1)를 이용해 요청한 데이터를 보내준다. </br>

</details>

<details><summary>4. JWT 의 한계와 보안 전략 (Refresh Token)</summary> </br>

❶ 치명적 약점: “통제 불능의 공포” </br>

세션 방식에서는 해커가 세션 ID 를 훔쳐 가도, 서버 관리자가 해당 세션을 Redis 에서 지워버리면(강제 로그아웃) 방어가 가능했다. </br>

하지만 JWT 는 다르다. </br>

한 번 발급되어 클라이언트에게 넘어간 토큰은, 유효기간이 끝날 때까지 서버가 강제로 죽일 방법이 없다. </br>

📄 시나리오: 해커가 사용자의 JWT 를 탈취했다. </br>

🤔 서버의 입장: “어? 저 토큰 해커가 쓰고 있는 것 같은데… 근데 서명(Signature)도 맞고 유효기간도 남았네? 막을 명분이 없다. 통과!” </br>

결과: 해커는 토큰 수명이 다할 때까지 사용자 행세를 하며 활개친다. </br>

</br>

❷ 유효기간(Expiration)의 딜레마 </br>

이 문제를 해결하기 위해 가장 먼저 건드리는 것이 바로 exp(만료 시간)이다. 하지만 여기서 보안과 사용자 경험(UX) 사이의 줄다리기가 시작된다. </br>
 
      1. 짧게 설정한다 (예: 5분) </br>
      
      - 보안: 해커가 훔쳐도 5분 뒤면 무용지물이 된다. (안전함) </br>
      
      - UX: 사용자는 5분마다 로그인이 풀려서 다시 아이디와 비밀번호를 입력해야 한다. (최악의 경험) </br>
      
         </br>
         
      2. 길게 설정한다 (예: 1주일) </br>
      
      - UX: 한 번 로그인하면 1주일간 편하게 쓴다. (편리함) </br>
      
      - 보안: 해커에게 1주일짜리 프리패스를 주는 꼴이다. (위험함) </br>
      
      “보안도 챙기고, 사용자도 편하게 할 수는 없을까?” </br>
      
      이 난제를 해결하기 위해 등장한 것이 바로 이중 토큰(Two Token) 전략이다. </br>

✏️ 해결책: Access Token과 Refresh Token의 공존 </br>

단 하나의 토큰으로 모든 걸 하려다 보니 문제가 생긴다. 그래서 용도를 둘로 나눈다. </br>

“하나는 자주 쓰는 대신 빨리 죽이고, 하나는 가끔 쓰는 대신 오래 살려두자.” </br>

1. Access Token (접근 토큰) </br>

- 용도: 실제 API 데이터를 요청할 때 사용하는 출입증. </br>

- 수명: 짧음 (30분 ~ 1시간). </br>

- 특징: 탈취당해도 수명이 짧아 피해가 제한적이다. 우리가 지금까지 배운 Stateless JWT 가 바로 이것이다. </br>

2. Refresh Token (재발급 토큰) </br>

- 용도: Access Token 이 만료됐을 때, “새것으로 교환해 주세요” 라고 할 때만 사용하는 교환권. </br>

- 수명: 아주 김 (2주 ~ 1달). </br>

- 특징: 평소에는 요청에 실어 보내지 않으므로 탈취 위험이 적다. </br>

- 중요: 이 토큰은 서버의 DB(또는 Redis)에 저장하여 관리한다. </br>

🌟 왜 이렇게 하는가?: 보안과 편의성을 다 잡기 위해서다. Access Token을 짧게 가져가서 보안을 지키고, </br>
Refresh Token으로 사용자가 매번 다시 로그인하는 불편함을 없앤다. </br>

<details><summary>5. 암호화의 기초: 비밀번호 저장 방식 (Security)</summary> </br>

인증의 시작은 로그인이고, 로그인의 핵심은 "사용자의 비밀번호를 어떻게 보관하느냐"이다. </br>

백엔드 개발자가 절대 해서는 안 되는 행동 1순위는 DB에 비밀번호를 그대로(Plain Text) 저장하는 것이다. </br>

❶ 단방향 해시 함수 (One-way Hash) </br>

- 원리: 어떤 문장을 넣어도 고유한 암호문으로 바꿔주는 함수이다. </br>

- 특징: 암호문을 다시 원문으로 되돌리는 것(복호화)이 불가능하다. </br>

- 백엔드의 검증: 서버도 유저의 진짜 비밀번호를 모른다. 유저가 로그인할 때 입력한 값을 똑같이 해싱해서, DB에 저장된 암호문과 일치하는지만 확인한다. </br>

</br>

❷ 솔팅 (Salting) </br>

- 문제점: 똑같은 비밀번호(예: 1234)를 쓰는 유저가 많으면, DB에 똑같은 해시값이 저장되어 해커가 유추하기 쉬워진다. </br>

- 해결책: 비밀번호 뒤에 무작위 문자열(Salt, 소금)을 살짝 뿌려 해싱하는 기법이다. </br>

- 결과: 똑같은 '1234'라도 사람마다 암호문이 완전히 다르게 생성된다. </br>

</br>

❸ Bcrypt 알고리즘 </br>

- 설명: 현재 백엔드 실무에서 가장 많이 사용되는 비밀번호 암호화 방식이다. </br>

- 장점: 내부적으로 솔팅(Salting)을 알아서 처리해 준다. 또한 해커가 무차별 대입을 하지 못하도록 연산 속도를 의도적으로 늦추는 기능을 제공한다. </br>

</details>

<details><summary>6. 최종 요약: 세션(Session) vs 토큰(JWT)</summary> </br>

| 구분 | 세션 (Session) | 토큰 (JWT) |
| :--- | :--- | :--- |
| **데이터 성격** | **참조 (Reference)** | **값 (Value / Self-contained)** |
| **비유** | **은행 번호표** | **여권** |
| **동작 원리** | 번호표 종이 자체엔 아무 정보가 없음. 은행원이 창구 모니터(서버 DB)를 봐야만 누구인지 알 수 있음. | 여권 자체에 사진, 이름, 국적(데이터)이 다 적혀 있음. 출입국 심사관은 여권의 위조 여부만 확인하면 됨. |
| **상태 저장** | **Stateful** (서버 메모리/Redis 사용) | **Stateless** (서버 저장소 불필요) |
| **검증 비용** | 매 요청마다 DB/Redis I/O 발생 | CPU 연산(서명 확인)만 발생 |

</details>

**Q1. 인증(Authentication)과 인가(Authorization)의 차이를 설명해 주세요** </br>

답변: "인증은 '당신이 누구인지' 신원을 확인하는 단계(예: 로그인)이고, </br> 
인가는 인증된 사용자가 '특정 자원에 접근할 권한이 있는지' 확인하는 단계(예: 관리자 페이지 접근 권한)입니다. </br>
호텔에 비유하자면 체크인을 통해 신분증을 확인하는 것이 인증이고, 발급받은 키카드로 내 방에만 들어갈 수 있게 제한하는 것이 인가입니다." </br>

</br>

**Q2. JWT(JSON Web Token)의 구조에 대해 설명해 주세요.** </br>

답변: "JWT는 점(.)을 구분자로 하여 Header, Payload, Signature 세 부분으로 구성됩니다. </br>
Header는 토큰 타입과 암호화 알고리즘 정보를 담고 있고, Payload는 유저 ID나 권한 같은 실제 데이터(Claim)를 담습니다. </br>
가장 중요한 Signature는 서버만 알고 있는 'Secret Key'를 이용해 생성된 암호문으로, 토큰의 위변조 여부를 검증하는 데 사용됩니다." </br>

</br>

**Q3. JWT는 Stateless하다고 하는데, 이게 왜 장점인가요?** </br>

답변: "서버가 사용자의 상태를 메모리(세션)나 DB에 저장할 필요가 없기 때문입니다. </br>
서버는 토큰의 서명(Signature)만 확인하면 즉시 유효한 사용자인지 알 수 있습니다. </br>
이 덕분에 서버를 여러 대 운영하는 환경에서 별도의 세션 동기화 작업 없이도 수평적 확장(Scale-out) 이 매우 용이하다는 것이 큰 장점입니다." </br>

</br>

**Q4. JWT를 사용하면서 발생할 수 있는 보안 문제는 무엇이며, 어떻게 해결하나요?** </br>

답변: "가장 큰 문제는 토큰을 탈취당했을 때 서버가 이를 강제로 무효화할 방법이 없다는 점입니다. </br>
이를 해결하기 위해 실제 서비스에서는 Access Token의 유효기간을 매우 짧게(예: 30분) 설정하고, </br>
토큰을 재발급받을 수 있는 Refresh Token을 별도로 운영합니다. </br>
만약 보안 사고가 의심되면 서버 DB에서 Refresh Token을 삭제하여 해커의 재발급을 차단할 수 있습니다." </br>

</br>

**Q5. Access Token은 클라이언트에 전송하지만, Refresh Token은 왜 서버 DB에 저장하나요?** </br>

답변: "JWT의 단점인 '통제 불가능성'을 보완하기 위해서입니다. </br>
Access Token은 Stateless의 장점을 살려 서버 조회 없이 빠르게 인증에 사용하지만, </br>
Refresh Token은 서버가 직접 관리(저장)함으로써 사용자의 로그아웃을 강제하거나, </br>
탈취된 토큰을 무효화하는 최소한의 '통제권' 을 갖기 위함입니다." </br>

</br>

**Q6. 비밀번호를 DB에 저장할 때 해시(Hash) 함수를 사용하는 이유는 무엇인가요?** </br>

답변: "비밀번호는 보안상 절대로 평문으로 저장해서는 안 되기 때문입니다. </br>
해시 함수는 단방향성을 가지므로, 설령 DB가 해킹당하더라도 해커가 암호문을 원래의 비밀번호로 되돌릴 수 없습니다. </br>
여기에 무작위 문자열을 추가하는 솔팅(Salting) 기법을 더하면, 동일한 비밀번호라도 사용자마다 다른 해시값이 생성되어 보안성을 더욱 높일 수 있습니다." </br>
