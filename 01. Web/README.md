# 🌐 01. WEB

<details>
<summary>1. Web이란?</summary> </br>

웹(Web)은 클라이언트(Client)와 서버(Server)가 HTTP를 통해 서로 통신하는 서비스 구조이다.</br>

사용자가 브라우저에서 특정 URL을 입력하면, 브라우저는 서버에 `요청(Request)`을 보낸다.</br>

서버는 해당 요청을 처리한 뒤 결과값을 `응답(Response)` 형태로 다시 반환한다.</br>

즉, 웹은 **요청(Request) → 응답(Response)** 구조로 동작한다.

</details>

<details>

**1) 클라이언트(Client)** </br>

클라이언트는 사용자가 웹 내 정보를 탐색할 수 있게 도와주는 역할을 한다. </br>

우리 주변에서 쉽게 볼 수 있는 예시로는 Google Chrome, Safari 등이 있다. </br>

클라이언트는 `웹브라우저` 로도 불린다. </br>

즉, 우리가 웹페이지에 접근할 수 있게 도와주는 소프트웨어라고 생각하면 된다. </br>

브라우저는 사용자가 입력한 URL을 통해 서버(Server)에 요청을 보낸다. </br>

</br>

**2) 서버(Server)** </br>

웹 서버는 클라이언트의 요청을 처리하고 처리 결과를 `응답(Response)으로 반환`하는 역할을 한다. </br>

쉽게 말하자면 사용자에게 필요한 정보(웹사이트, 데이터 등)를 제공(Serve)해주는 역할이다. </br>

대표적인 웹 서버에는 Apache, Nginx 등이 있다. </br>

</br>

**3) 웹 프로토콜** </br>

웹 프로토콜이란 클라이언트와 서버가 데이터를 주고받는 `통신 규칙`을 말한다. </br>

웹이 원활하게 작동하기 위해선 여러 프로토콜이 필요한데 이러한 프로토콜은 데이터 전송의 규칙을 정의하는 역할을 한다.</br>

</details>

<details>
<summary>3. HTTP </summary> </br>

1) HTTP의 정의 </br>

클라이언트와 서버 간의 데이터를 주고 받기 위한 프로토콜이다. </br>

</br>

2) HTTP의 특징 </br>

- 요청 - 응답 구조 </br>

보통 클라이언트에 의헤 전송되는 메세지를 요청(Request)라고 부르며 서버에서 응답하는 메세지를 응답(Response)라고 부른다. <br>

클라이언트 → Request → 서버 → Response → 클라이언트 순으로 진행된다.

- 무상태 </br>

HTTP는 이전의 상태를 기억하지 않는다. </br>

따라서 쿠키 또는 세션을 이용해 정보를 저장해야한다. </br>

쿠키란 `클라이언트(브라우저)에 저장되는 데이터` 를 뜻하며, </br>

세션은 `서버에 저장되는 사용자 정보`를 뜻한다. </br>

- 비연결성 (Connectionless) </br>

HTTP는 요청과 응답이 끝나면 연결을 종료한다. 이로 인해 서버 자원을 효율적으로 사용할 수 있다. </br>
 
다만, 매번 연결을 새로 맺는 비용(Handshake)을 줄이기 위해 실제로는 Keep-Alive 옵션을 통해 일정 시간 연결을 유지하기도 한다. </br>

</br>

3) HTTP 메서드 </br>

HTTP 메서드에는 GET, POST, PUT, DELETE, PATCH 등이 있다. </br>

- GET </br>

사용자가 웹 페이지에 접근할 때 GET 요청이 발생한다. </br>

GET은 서버로부터 어떠한 데이터를 가져오기 위해 사용된다. </br>

여기서 중요한 점은,

`URL에 넘겨주는 값이 그대로 노출된다`는 특징이 있어 보안이 약하므로, 민감한 데이터 전송에는 적합하지 않다. </br>

- POST </br>

POST 요청은 사용자가 웹 페이지에서 데이터를 제출할 때 사용된다. </br>

POST 요청은 보통 HTML 양식을 통해 서버에 전송되며, GET 방식과 달리 서버상의 `데이터 값이나 내용 등을 바꾸기 위해` 사용된다. </br>

사용자가 웹 페이지 내에서 정보를 입력한 후, 데이터를 제출하면 이 정보는 POST 방식을 통해 서버에 전달되는 것이다. </br>

또한 GET 방식과 달리 POST 방식은 값을 전달할 시 데이터를 Body에 담아 전송한다. </br>

하지만 HTTP 자체는 암호화되지 않기 때문에, 보안은 HTTPS 사용 여부에 따라 결정된다. </br>

- PUT </br>

리소스를 `전체 재수정할 때` 사용된다. </br>

기존 데이터를 완전히 덮어버린다는 특징이 있으며, 사용자 정보를 전체 수정하거나 게시글 전체 수정이 필요한 경우 주로 사용된다. </br>

- PATCH </br>

PUT과 달리 PATCH는 리소스를 `부분 수정할 때` 사용된다. </br>

비밀번호 변경, 닉네임 변경 등 일부 필드만 수정할 때 주로 쓰인다. </br>

- DELETE </br>

지정된 리소스를 `삭제할 때` 사용된다. <br>

DB에서 완전히 지워버리는 Hard Delete 와 데이터를 삭제하지 않고 삭제 표시만 남기는 Soft Delete 가 있다.</br>

복구가 불가능한 Hard Delete와 달리 Soft Delete는 복구가 가능해, 실무에서는 데이터 복구 및 기록 보존을 위해 Soft Delete를 많이 사용한다고 한다.  

🌟 PUT/PATCH/GET/DELETE는 여러 번 실행해도 결과가 같은 **'멱등성'** 을 가지지만, POST는 매번 새로운 리소스를 생성할 수 있어 멱등하지 않다. </br>

</br>
   
4) HTTP 상태 코드 </br>

HTTP 상태 코드는 서버가 클라이언트의 요청을 처리한 결과를 숫자로 나타낸 것이다. </br>

상태 코드는 크게 5가지 범주로 나뉜다. </br>

</br>

1️⃣ 1xx (정보응답) </br>
요청을 받았으며 처리 중임을 의미한다. </br>

- 100 Continue : 클라이언트가 요청을 계속해도 된다는 의미.

2️⃣ 2xx (성공) </br>
요청이 정상적으로 처리되었음을 의미한다. </br>

- 200 OK : 요청이 성공적으로 처리되었다. </br>

- 201 Created : 요청이 성공적으로 처리되었으며 새로운 리소스가 생성되었다. </br>

- 204 No Content : 성공 상태 응답 코드는 요청이 성공했으나 응답 바디 값이 없다. <br>

3️⃣ 3xx (리다이렉션) </br>
요청을 완료하기 위해 추가 동작이 필요하다. </br>

- 301 Moved Permanently : 요청한 리소스가 헤더에 주어진 URL로 완전히 옮겨졌다. </br>

- 302 Found : 요청한 리소스가 헤더에 주어진 URL로 일시적으로 옮겨졌다. </br>

4️⃣ 4xx (클라이언트 오류) </br>
요청의 문법이 잘못되었거나 요청을 처리할 수 없다. </br>

- 400 Bad Request : 잘못된 요청이다. </br>

- 401 Unauthorized : 인증이 필요하다. </br>

- 403 Forbidden : 서버가 요청을 받았지만 접근 권한이 없어 승인을 거부한다. </br>

- 404 Not Found : 서버가 요청 받은 리소스를 찾을 수 없다. </br>

5️⃣ 5xx (서버 오류) </br>
서버 내부의 문제로 요청 처리에 실패하였다. </br>

- 500 Internal Server Error : 요청을 처리하는 과정에서 서버가 예상하지 못한 상황에 놓였다. </br>

- 503 Service Unavailable : 서버가 요청을 처리할 준비가 되지 않았다. 보통 서버가 점검을 위해 다운되거나 과부하 됐을 때 발생한다. </br>

</br>

5) URL </br>

URL(Uniform Resource Locator)은 인터넷 상에서 리소스의 위치를 나타내는 주소를 뜻한다. </br>

클라이언트는 URL을 통해 서버의 특정 자원(Resource)에 접근한다. </br>

예시)
`https://www.example.com:8080/users/1?name=kim` </br>

이 예시에서 본다면, </br>

`https` -> 어떤 통신 규칙을 사용할지 나타내주는 `프로토콜`에 해당한다. </br>

`www.example.com` -> 해당 서버의 `주소`를 의미한다. </br>

`8080` -> 서버 내부의 특정 프로그램과 연결하기 위한 번호. 즉, `포트 번호`를 의미한다. </br>

`/users/1` -> 서버 내에서 접근하려는 리소스. 즉, `경로`를 의미한다. </br>

`?name=kim` -> 추가적인 데이터를 전달할 때 사용되는 `쿼리 스트링`을 의미한다. 
주로 검색, 정렬, 필터링 등에 사용된다. </br>

* URL과 URI 의 차이점 </br>

- URI : 웹 내의 자원을 식별하는 모든 방법을 뜻한다. </br>

우리가 흔히 알고 있는 URL을 포함하는 더 큰 개념이다. </br>

- URL : 위치가 포함된 식별자를 뜻한다. </br>

http, https 등을 사용하여 어떤 위치에 있는지에 관한 정보를 포함해, 실제로 우리가 브라우저에 입력하는 주소이기도 하다. <br>

</details>

<details>
<summary>4. HTTPS </summary> </br>
   
HTTPS는 HTTP에 `암호화 계층(SSL/TLS)이 추가`된 프로토콜이다. </br>

HTTPS는 아래와 같은 특징이 있다. </br>

- 데이터를 암호화하여 중간 탈취를 방지한다. </br>

- 인증서를 통해 서버의 신뢰성을 보장한다. </br>

- HTTPS는 기본적으로 443 포트를 사용한다. </br>
   
</details> 

<details>
<summary>🔥 핵심 면접 질문 리스트 🔥 </summary></br>

**Q1. HTTP와 HTTPS의 차이점은 무엇인가요?** </br>

답변: "가장 큰 차이는 **보안(암호화)** 입니다. <br>
HTTP는 텍스트를 그대로 전송하기 때문에 중간에 데이터를 가로챌 위험이 크지만, </br>
HTTPS는 SSL/TLS 프로토콜을 추가해 데이터를 암호화합니다. </br>
또한 HTTP는 80번 포트를, HTTPS는 443번 포트를 사용합니다. <br>
최근엔 구글 검색 순위 가산점이나 사용자 데이터 보호를 위해 모든 사이트가 HTTPS를 사용하는 것이 표준입니다." </br>

</br>

**Q2. GET과 POST의 차이점을 설명해 보세요.** </br>

답변: "데이터를 다루는 목적과 방식이 다릅니다. </br>
GET은 데이터를 '조회'하기 위해 사용하며, 데이터가 URL 뒤에 쿼리 스트링으로 붙습니다. <br>
결과가 변하지 않는 멱등성을 가지며 브라우저 캐싱이 가능합니다. </br>
반면 POST는 데이터를 '생성'하거나 '변경'할 때 사용하며, 데이터가 HTTP Body에 담겨 전송됩니다. <br>
호출할 때마다 새로운 데이터가 생길 수 있어 멱등하지 않습니다." </br>

</br>

**Q3. PUT과 PATCH의 차이는 무엇인가요?** </br>

답변: "수정하는 범위의 차이입니다. </br>
PUT은 해당 자원의 '전체 수정'을 의미합니다. <br>
만약 일부 필드만 보내면 나머지 필드는 기본값이나 null로 덮어씌워질 수 있습니다. </br>
반면 PATCH는 '부분 수정'을 의미하며, 보낸 필드 값만 업데이트합니다. <br>
실무에서는 닉네임 하나만 바꿀 때처럼 자원을 아끼고 싶을 때 PATCH를 주로 사용합니다." </br>

</br>

**Q4. HTTP의 '무상태(Stateless)'란 무엇이며, 어떻게 보완하나요?** </br>

답변: "서버가 클라이언트의 이전 상태를 기억하지 않는 성질입니다. </br>
덕분에 서버가 바뀌어도 상관없어 확장성이 좋지만, 로그인 유지 같은 기능이 불가능합니다. </br>
이를 보완하기 위해 클라이언트 측에 저장하는 쿠키(Cookie), 서버 메모리에 저장하는 세션(Session), </br>
혹은 서버에 저장하지 않고 검증만 하는 JWT 토큰 방식을 사용하여 사용자의 로그인 상태를 식별합니다." </br>

</br>

**Q5. 401(Unauthorized)과 403(Forbidden) 에러의 차이는 무엇인가요?** </br>

답변: "쉽게 말해 **'인증'** 과 **'인가'** 의 차이입니다. </br>
401은 로그인이 되지 않았거나 유효하지 않은 계정일 때 발생하는 '인증 실패' 오류입니다. </br>
반면 403은 로그인은 성공해서 누구인지는 알겠으나, 관리자 페이지처럼 특정 리소스에 접근할 권한이 없는 '인가 실패' 오류입니다." </br>

</br>

**Q6. 주소창에 www.google.com을 입력하면 어떤 일이 일어나나요?** </br>

답변: "크게 5단계로 일어납니다. </br>
먼저 `1. DNS를 통해 도메인 이름을 실제 IP 주소로 변환합니다.` </br>
그 후 `2. TCP 3-Way Handshake를 통해 서버와 연결을 맺습니다.` </br>
연결이 되면 `3. HTTP 요청을 서버로 보내고, 서버는 해당 정보를 처리하여` `4. HTTP 응답을 보냅니다.` </br>
마지막으로 브라우저는 받은 HTML, CSS, JS를 해석하여 사용자가 보는 화면을 그리는 `5. 렌더링 과정을 거칩니다.`"

</details>
